# Practical 4: Windows Log Analysis

## Overview

In this practical, I explored how Windows records security related events and how those events can be reviewed using Event Viewer.

The primary focus of this exercise was analyzing failed login attempts through the Windows Security Log and understanding how security analysts use authentication events during investigations.

The practical was performed inside a Windows 10 virtual machine. No system settings, registry values, or security configurations were modified. The objective was to safely examine existing logs and understand the investigation process used in real-world environments.

## Tools Used

### Event Viewer

Event Viewer is a built-in Windows monitoring tool used to review system, application, and security related events.

For this practical, Event Viewer was used to:

* Access Windows Security Logs
* Filter authentication events
* Review failed login attempts
* Examine event details relevant to investigations

---

## Practical Steps

### Step 1: Open Event Viewer

The practical started by opening Event Viewer from the Start Menu inside the Windows 10 virtual machine.

From Event Viewer, I expanded the **Windows Logs** section to review the available log categories.

The following log categories were observed:

| Log Type    | Purpose                                    |
| ----------- | ------------------------------------------ |
| Application | Events generated by applications           |
| Security    | Authentication and security related events |
| Setup       | Installation and configuration events      |
| System      | Operating system events                    |

For authentication investigations, the Security Log is the most important category because it records login activity and other security-related actions.

---

## Step 2: Access the Security Log

After opening the Security Log, Windows displayed a large number of recorded security events.

Each event contained a unique Event ID that identifies a specific activity.

At this stage, the Security Log was reviewed to understand how authentication-related events are recorded and organized.

---

## Step 3: Locate Failed Login Attempts

To identify failed login activity, the Security Log was filtered using:

**Event ID: 4625**

Event ID 4625 is generated whenever Windows records a failed authentication attempt.

After applying the filter, multiple failed login events were identified and reviewed.

---

## Step 4: Review Event Details

Each failed login event contained useful information that helps analysts understand what happened during the authentication attempt.

The following fields were examined:

| Field          | Purpose                                             |
| -------------- | --------------------------------------------------- |
| Time Created   | Indicates when the failed login occurred            |
| Account Name   | User account involved in the authentication attempt |
| Failure Reason | Explains why authentication failed                  |
| Logon Type     | Indicates how the login attempt was performed       |

These fields provide valuable context during authentication investigations.

---

## Step 5: SOC-Level Investigation Approach

Finding a failed login event is only the beginning of an investigation.

A SOC analyst typically reviews multiple events and looks for patterns rather than focusing on a single entry.

During this practical, the investigation process included:

* Reviewing multiple Event ID 4625 entries
* Comparing timestamps
* Examining targeted accounts
* Reviewing authentication details
* Looking for repeated failures

This approach helps determine whether activity appears normal or requires additional investigation.

---

## Indicators of Suspicious Activity

While reviewing authentication logs, analysts often look for patterns such as:

* Repeated failed login attempts against the same account
* Multiple accounts experiencing authentication failures
* Unusual login times
* Authentication activity originating from unexpected sources
* High volumes of failed logins within a short period

It is important to note that failed login attempts do not automatically indicate malicious activity. Users can enter incorrect passwords or trigger authentication failures for legitimate reasons.

The surrounding context must always be considered before drawing conclusions.

---

## Investigation Workflow

The workflow followed during this practical was:

1. Open Event Viewer
2. Navigate to Windows Security Logs
3. Filter logs using Event ID 4625
4. Review authentication event details
5. Analyze timestamps and account activity
6. Look for patterns across multiple events
7. Determine whether additional investigation may be required

This process reflects the basic methodology commonly used during authentication-related investigations.
